# ТЕХНИЧЕСКОЕ ЗАДАНИЕ (Brief)

**Название модуля:** Модуль «Карта социальных рисков» для ИС «Протокол» в рамках ОПиСЭР.

**Расположение в системе:** раздел «ОПИСЭР» → **«Карта социальных рисков»**. Контент модуля соответствует **стр. 20–23** презентации (слайды Шымкент 2025).

**Ответственный за заполнение данных:** Департамент полиции (пользователи с ролью **«ДП»**).

---

## 1. Цель и назначение

Ведение карты социальных рисков: сбор, хранение и отображение данных по социальным рискам на территории (районы, населённые пункты, показатели, уровни риска, принятые меры). Данные в модуль вносит **департамент полиции**. Модуль обеспечивает автоматическую генерацию слайдов по карте социальных рисков (стр. 20–23) в форматах **PPTX** и **PDF** для отчётности, идентичных по структуре загруженному образцу. Контент слайдов заполняется в системе; выгрузку выполняет администратор протокола.

---

## 2. Ролевая модель и доступ

В модуле используются две роли.

**ДП (Департамент полиции):**
- Пользователь, наделённый ролью **«ДП»**, заполняет и редактирует данные модуля «Карта социальных рисков».
- Видит только данные этого модуля (в рамках своих полномочий). Не выполняет выгрузку в PPTX/PDF.

**Администратор протокола (глобальный администратор):**
- Имеет доступ на чтение ко всем данным карты социальных рисков.
- Имеет право **выгрузки** слайдов по карте (стр. 20–23) в PPTX или PDF. Выгрузку выполняет только пользователь с ролью **«Администратор протокола»**.
- Не заполняет данные модуля (данные вносит ДП); при необходимости может видеть дату последнего обновления и историю изменений (если реализовано).

---

## 3. Структура данных и соответствие слайдам

Контент слайдов **стр. 20–23** формируется из данных, введённых в системе. Точный состав полей и вид блоков на экране необходимо согласовать с макетом слайдов (Слайды Шымкент 2025, стр. 20–23). Ниже приведена типовая структура; при наличии готового макета слайдов её нужно уточнить.

### 3.1. Общие поля (период, дата обновления)

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| **Период отчётности** | Дата / год и месяц (или квартал) | Да | Период, за который актуальна карта (например, месяц и год). Используется для фильтрации и подстановки в выгрузку. |
| **Дата последнего обновления** | Дата (системная) | — | Заполняется автоматически при сохранении. Отображается в интерфейсе. |

### 3.2. Блок для слайда 20 (стр. 20)

Слайд 20, как правило, — сводный или вводный (общая картина, ключевые показатели). Рекомендуемый тип данных — **сводные показатели или краткое описание**.

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| **Сводка / описание (стр. 20)** | Текст (многострочный) или структурированные поля | По макету | Текст, цифры или короткие показатели для первого слайда. Конкретный набор полей уточняется по макету слайда 20. |

*Вариант:* при наличии на слайде таблицы или списка показателей — реализовать блок с полями вида: «Показатель 1», «Значение 1», «Показатель 2», «Значение 2» и т.д., либо одну таблицу с колонками «Показатель» / «Значение» (редактируемые строки).

### 3.3. Блок для слайдов 21–23 (стр. 21, 22, 23)

Слайды 21–23 обычно содержат детализацию: по районам, по видам рисков, по территориям. Рекомендуемый тип данных — **записи (строки)** с общими полями.

**Вариант А — список записей «Территория / вид риска»:**

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| **Территория / район** | Текст или справочник | Да | Район, населённый пункт или иная территория. |
| **Вид риска (категория)** | Текст или справочник | Нет | Категория социального риска (при наличии справочника — выпадающий список). |
| **Уровень риска** | Справочник | Нет | Например: высокий / средний / низкий. |
| **Показатель** | Число или текст | Нет | Значение показателя (количество, доля и т.п.). |
| **Принятые меры / комментарий** | Текст | Нет | Краткое описание мер или примечание. |
| **Порядок отображения** | Число | Нет | Для сортировки при выводе на слайд. |

Каждая такая запись может относиться к одному из слайдов 21, 22 или 23 (поле **«Слайд»** со значениями 21, 22, 23), либо разнесение по слайдам задаётся правилом (например, по типу территории или по виду риска) — уточняется по макету.

**Вариант Б — три отдельных блока (по одному на слайд 21, 22, 23):**

- Для каждого слайда — свой набор полей или своя таблица. Состав полей берётся из макета соответствующей страницы презентации.

Итоговую схему данных (поля по каждому слайду, тип ввода — форма/таблица) необходимо зафиксировать на этапе проектирования по образцу слайдов 20–23.

### 3.4. Справочники (при необходимости)

- **Территории (районы, населённые пункты)** — при унификации названий.
- **Виды социальных рисков** — при категоризации.
- **Уровень риска** — высокий / средний / низкий (или по заданию заказчика).

## 4. Требования к интерфейсу

### 4.1. Размещение в системе

- В боковом меню раздела «ОПИСЭР» — пункт **«Карта социальных рисков»** (отдельно от «Реестр кадров»).
- При переходе открывается экран модуля с данными за выбранный период (или последний сохранённый период).

### 4.2. Экран редактирования (роль ДП)

- Выбор **периода отчётности** (год, месяц или квартал — по решению заказчика).
- **Блоки данных**, соответствующие стр. 20, 21, 22, 23:
  - для стр. 20 — форма с полями сводки/описания или таблица показателей (по макету);
  - для стр. 21–23 — таблица записей (территория, вид риска, уровень, показатель, меры) с возможностью добавления/удаления строк, либо три отдельные формы/таблицы.
- Кнопки **«Сохранить»**, при необходимости **«Отменить»**.
- Отображение **даты последнего обновления** данных.

### 4.3. Экран просмотра (роль ДП и Администратор протокола)

- Только чтение тех же блоков данных и периода. Для роли «Администратор протокола» — дополнительно кнопки выгрузки (см. ниже).

### 4.4. Выгрузка (роль Администратор протокола)

- Кнопки **«Выгрузить в PDF»** и **«Выгрузить в PPTX»** отображаются только у пользователей с ролью **«Администратор протокола»**.
- При выгрузке формируются слайды **стр. 20–23** по образцу презентации; содержимое подставляется из данных модуля за выбранный период (или текущий сохранённый период). Форматы: PPTX (приоритет) и PDF. Элементы в PPTX — редактируемые объекты (текст, таблицы, изображения), а не единая картинка.

---

## 5. Требования к функционалу выгрузки

- Выгрузка доступна **только пользователям с ролью «Администратор протокола»**.
- **Образец:** стр. 20–23 презентации «Слайды Шымкент 2025».
- **Форматы:** PPTX и PDF. Содержимое слайдов формируется из данных, введённых пользователем с ролью ДП.
- **PPTX:** сохранение вёрстки образца; текст и таблицы — редактируемые объекты для возможной ручной правки перед печатью.
- **Период:** в выгрузку попадают данные за выбранный период отчётности (или за период по умолчанию — последний сохранённый).

## 6. User Stories (пользовательские истории)

### Эпик 1: Заполнение данных (роль: ДП — Департамент полиции)

**US-K1.1 (Вход и редактирование):**  
Как пользователь с ролью ДП, я хочу открыть раздел «Карта социальных рисков», выбрать период отчётности и заполнить (или отредактировать) данные по стр. 20–23, чтобы актуальная карта социальных рисков была в системе и попадала в выгрузку слайдов.

**US-K1.2 (Сохранение):**  
Как пользователь с ролью ДП, я хочу нажать «Сохранить» и увидеть подтверждение, а также обновлённую дату последнего обновления, чтобы быть уверенным, что данные сохранены.

**US-K1.3 (Табличные данные):**  
Как пользователь с ролью ДП, я хочу добавлять, редактировать и удалять строки в таблицах (территории, показатели, меры), чтобы отражать изменения по районам и видам рисков.

### Эпик 2: Просмотр и выгрузка (роль: Администратор протокола)

**US-K2.1 (Просмотр карты):**  
Как администратор протокола, я хочу открыть «Карту социальных рисков» и просмотреть введённые ДП данные за выбранный период в режиме только чтения, чтобы проверить содержание перед выгрузкой.

**US-K2.2 (Выгрузка слайдов):**  
Как администратор протокола, я хочу нажать «Выгрузить в PPTX» (или PDF) и получить слайды стр. 20–23 с данными из системы, чтобы использовать их в отчётности без ручного копирования.

**US-K2.3 (Выбор периода для выгрузки):**  
Как администратор протокола, я хочу при выгрузке выбрать период отчётности (или использовать текущий), чтобы выгрузить нужный вариант карты.

### Эпик 3: Технические требования (для разработчиков)

**Tech-K1:** Реализовать роль **«ДП»** (Департамент полиции) и назначение прав: только модуль «Карта социальных рисков» на редактирование; без прав выгрузки.

**Tech-K2:** Реализовать хранение данных модуля по периодам отчётности (год/месяц или квартал) и отображение в интерфейсе выбранного периода.

**Tech-K3:** Реализовать генерацию слайдов стр. 20–23 в форматах PPTX и PDF на основе сохранённых данных модуля. Состав полей и разметку слайдов согласовать с макетом презентации.

**Tech-K4:** После согласования макета слайдов 20–23 — зафиксировать итоговую структуру полей (разделы, таблицы, справочники) в дополнении к настоящему ТЗ или в отдельном документе конфигурации модуля.

| Слайд | Источник данных в системе |
|-------|---------------------------|
| Стр. 20 | Блок данных «Стр. 20» (сводка/показатели) — поля уточняются по макету слайда 20. |
| Стр. 21 | Блок данных «Стр. 21» (таблица или форма) — поля уточняются по макету слайда 21. |
| Стр. 22 | Блок данных «Стр. 22» (таблица или форма) — поля уточняются по макету слайда 22. |
| Стр. 23 | Блок данных «Стр. 23» (таблица или форма) — поля уточняются по макету слайда 23. |

Конкретный состав полей для каждого слайда необходимо определить по презентации «Слайды Шымкент 2025» (стр. 20–23) и внести в итоговую спецификацию модуля.

## 8. Рекомендация для заказчика

**Уточнение полей по слайдам:**  
Текущее ТЗ задаёт общую логику модуля (роль ДП, выгрузка стр. 20–23, период отчётности). Окончательный перечень полей, заголовков таблиц и расположение блоков на экране целесообразно зафиксировать после просмотра слайдов 20–23: выписать с каждого слайда названия блоков, таблиц и показателей и перенести их в структуру данных и интерфейс модуля. Это позволит добиться полного соответствия выгрузки макету презентации.
